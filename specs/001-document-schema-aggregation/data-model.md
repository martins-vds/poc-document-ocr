# Phase 1 — Data Model

**Feature**: Consolidated Processed-Document Schema
**Storage**: Azure Cosmos DB SQL API, container `ProcessedDocuments`, partition key `/identifier`

This document defines the persisted JSON shape and the in-memory C# models. The container is wiped at deployment per FR-010 (D8 in `research.md`); no migration logic is required.

When discussing them in code review or commit messages, qualify as "record-level reviewedBy" or "field-level reviewedBy".

## Entity: `SchemaField`

Per-field OCR + reviewer state.

| Property (C#)   | JSON            | Type                       | Validation                                                                      |
| --------------- | --------------- | -------------------------- | ------------------------------------------------------------------------------- |
| `OcrValue`      | `ocrValue`      | `object?` (string or bool) | **Immutable** after creation by the mapper. Type matches the field catalog row. |
| `OcrConfidence` | `ocrConfidence` | `double?`                  | **Immutable** after creation. `null` allowed (e.g., field absent from OCR).     |
| `ReviewedValue` | `reviewedValue` | `object?`                  | Mutable while record is checked out by the actor; nullable.                     |
| `ReviewedAt`    | `reviewedAt`    | `DateTime?` (UTC)          | Set by `DocumentReviewService` on every save that mutates this field.           |
| `ReviewedBy`    | `reviewedBy`    | `string?`                  | UPN.                                                                            |
| `FieldStatus`   | `fieldStatus`   | `string` enum              | One of `Pending`, `Confirmed`, `Corrected`.                                     |

### Field-status state machine (FR-016)

```
                 user accepts OCR
   Pending  ───────────────────────▶  Confirmed
      │
      │  user edits and saves
      ▼
   Corrected
```

Once a field leaves `Pending`, transitions back to `Pending` are not permitted via the WebApp UI. (Re-running OCR for an existing record is out of scope.) Transitions between `Confirmed` ↔ `Corrected` are permitted: a user who first confirmed and then corrected will end in `Corrected` (or vice versa). The mapper rejects writes that violate the state machine.

### Validation rules

- `OcrValue` and `OcrConfidence` MUST NOT be modified after the record is first created (validated by `DocumentReviewService.ApplyEdits`; rejection raises `InvalidOperationException` per FR-014).
- `FieldStatus = Confirmed` requires `ReviewedValue == null` OR `ReviewedValue` deeply equals `OcrValue`.
- `FieldStatus = Corrected` requires `ReviewedValue != null` AND `!Equals(ReviewedValue, OcrValue)`.
- `FieldStatus = Pending` requires `ReviewedValue == null` AND `ReviewedAt == null` AND `ReviewedBy == null`.

## Record-level state machine

`reviewStatus` derives from per-field statuses on every save:

```
all 13 fields in {Confirmed, Corrected}  ──▶  Reviewed
                                              │
            any field in Pending              │
        ◀─── (no transition; UI bug if hit)   ▼
                                          (terminal in spirit; persists)
   Pending ◀──────────────────────────  ──┘
   (initial; while any field is Pending)
```

- Initial state on creation by the mapper: `Pending` (all fields are `Pending`).
- Transition `Pending → Reviewed` happens implicitly during a save when the last `Pending` field is resolved. At that exact transition `ReviewedBy` and `ReviewedAt` are set and become immutable (FR-018).
- The reverse transition is not exposed by the WebApp.
